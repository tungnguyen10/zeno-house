# Zeno House - Project Status

Last reviewed from source: 2026-06-16.

Zeno House is an authenticated operations tool for managing rental buildings. The current codebase is no longer a starter app: it has end-to-end building, room, tenant, contract, service, meter-reading, and monthly billing flows.

## Current Stack

| Layer | Current implementation |
| --- | --- |
| App framework | Nuxt 4 compatibility mode, Vue 3, TypeScript strict |
| UI | TailwindCSS, operational dark theme, self-hosted Inter, `nuxt-svgo` icons |
| State | Pinia for auth/session; composables for server state and form workflows |
| Backend | Nuxt server routes, service/repository layers, Supabase Postgres/Auth |
| Validation | Zod schemas in `app/utils/validators/**` |
| Data mapping | DTO mappers in `app/utils/mappers/**` |
| Tests | Vitest with happy-dom, service/repository/unit/component coverage |
| Workflow docs | OpenSpec specs in `openspec/specs/**`; archived changes in `openspec/changes/archive/**` |

## Active Product Surface

### Dashboard

- Route: `/`
- API: `GET /api/dashboard/summary`
- Shows total buildings, rooms by status, active contracts, and tenants.
- Shape is guarded by `tests/utils/dashboard-summary-shape.test.ts`.

### Buildings

- Routes: `/buildings`, `/buildings/create`, `/buildings/[id]`, `/buildings/[id]/edit`, `/buildings/[id]/settings`, `/buildings/[id]/meter-readings`, `/buildings/[id]/rooms/[room]`
- API: `GET/POST /api/buildings`, `GET/PATCH/DELETE /api/buildings/[id]`, `GET /api/buildings/[id]/rooms/[room]`
- Includes slug support, operational config, room list, service settings, service matrix, and meter-reading workspace.
- Building identifiers are resolved by id or slug in server repositories where supported.

### Rooms

- Routes: `/rooms`, `/rooms/create`, `/rooms/[id]`, `/rooms/[id]/edit`, plus building-scoped room detail routes.
- API: `GET/POST /api/rooms`, `GET/PATCH/DELETE /api/rooms/[id]`
- Statuses: `available`, `occupied`, `maintenance`.
- Contract create/terminate flows update room occupancy unless the room is in maintenance.

### Tenants

- Routes: `/tenants`, `/tenants/create`, `/tenants/[id]`, `/tenants/[id]/edit`
- API: `GET/POST /api/tenants`, `GET/PATCH/DELETE /api/tenants/[id]`
- Includes enriched profile fields: gender, occupation, ID issue date/place, emergency contact name/phone.

### Contracts

- Routes: `/contracts`, `/contracts/create`, `/contracts/[id]`, `/contracts/[id]/edit`
- API: `GET/POST /api/contracts`, `GET/PATCH/DELETE /api/contracts/[id]`
- Nested API: occupants, payments, renewals, and contract services.
- Contract detail includes commercial terms, occupants/roommates, payments, renewals, service overrides, and handover readings.
- `contract_code` is now generated by migration and used in operational routes when available.

### Services

- API: `GET /api/service-catalog`
- Building services: `GET/POST /api/building-services`, `PATCH /api/building-services/[id]`
- Contract services: `GET /api/contract-services`, `GET /api/contract-services/by-building`, `PATCH /api/contract-services/[id]`, `POST /api/contract-services/sync`
- Catalog contains the standard recurring services. Building settings define defaults; contracts clone and can override them.

### Meter Readings

- API: `GET/POST /api/meter-readings`, `PATCH /api/meter-readings/[id]`, `GET/POST /api/meter-readings/bulk`
- Model uses `(room_id, meter_type, period_year, period_month, reading_type)`.
- Reading types: `monthly`, `handover_in`, `handover_out`.
- Monthly billing can use previous monthly readings, handover fallback readings, or explicit utility usage overrides.

### Monthly Billing

- Routes: `/billing`, `/billing/[building]/[period]`, `/billing/invoices/[id]`
- Period API:
  - `GET/POST /api/billing/periods`
  - `GET /api/billing/periods/[id]`
  - `GET /api/billing/periods/[id]/overview`
  - `GET /api/billing/periods/[id]/drafts`
  - `GET /api/billing/periods/[id]/draft-grid`
  - `POST /api/billing/periods/[id]/utility-usages`
  - `GET /api/billing/periods/[id]/utility-usages`
  - `POST /api/billing/periods/[id]/issue`
  - `GET /api/billing/periods/[id]/invoices`
  - `GET /api/billing/periods/[id]/audit`
  - `GET /api/billing/periods/[id]/export`
  - `POST /api/billing/periods/[id]/close`
  - `POST /api/billing/periods/[id]/unissue`
- Invoice API:
  - `GET /api/billing/invoices/[id]`
  - `GET/POST /api/billing/invoices/[id]/payments`
  - `POST /api/billing/invoices/bulk-payments`
  - `POST /api/billing/invoices/[id]/adjustment`
  - `POST /api/billing/invoices/[id]/void`
  - `POST /api/billing/invoices/[id]/reissue`
- Workspace tabs:
  - Draft grid: enter readings, paste/bulk-save readings, review blockers, inspect discrepancy callouts.
  - Issue: issue eligible draft invoices.
  - Payments: collect payments, bulk collect, adjust, void/reissue.
- Header overflow actions: audit drawer, Excel export, close period, unissue period.
- Invoice detail route shows charges and payment history by `invoice_code` or id.

## Billing Rules In Source

- Period statuses: `draft`, `readings`, `review`, `issued`, `collecting`, `closed`.
- Invoice statuses: `draft`, `issued`, `partial`, `paid`, `overdue`, `void`.
- Charge types: `rent`, `electricity`, `water`, `service`, `discount`, `surcharge`, `adjustment`.
- Issuing recomputes drafts server-side and skips contracts that already have a non-void invoice for the period.
- Void is blocked when an invoice has any paid amount.
- Reissue creates a fresh invoice from the latest draft and links it to the voided invoice.
- Adjustment is used for correction after collection has started; negative adjustments cannot exceed paid amount.
- Closing requires `billing.close`, status `issued` or `collecting`, and no outstanding invoices.
- Unissue requires `billing.unissue`, a reason of at least 10 characters, and is blocked once the period is closed. It voids unpaid invoices and retains invoices with payments.
- Destructive billing actions write audit entries.

## Database Snapshot

There are 31 SQL migration files under `supabase/migrations`.

Important migration groups:

- Core property model: buildings, rooms, tenants.
- Contract model: contracts, commercial terms, occupants, payments, renewals.
- Service model: service catalog, building services, contract services.
- Meter model: simplified meter readings and handover/monthly reading types.
- Billing runtime: billing periods, invoices, invoice charges, invoice payments, utility usage overrides, audit events.
- Routing identifiers: building slugs plus contract/invoice codes.

Billing runtime tables have RLS policies in migration SQL. Application writes still go through API/service permissions rather than direct client Supabase calls.

## Server Architecture

The app uses the same shape across domains:

```text
server/api/*
  -> validates input, requires auth, returns envelope
server/services/*
  -> business rules, permissions, orchestration, audit
server/repositories/*
  -> Supabase query/persistence only
app/utils/mappers/*
  -> DB row to DTO
app/utils/validators/*
  -> shared Zod schemas
```

Primary server domain groups:

- `buildings`
- `rooms`
- `tenants`
- `contracts`
- `contract-occupants`
- `contract-payments`
- `contract-renewals`
- `service-catalog`
- `building-services`
- `contract-services`
- `meter-readings`
- `dashboard`
- `billing`

## Client Architecture

Composables mirror product workflows:

- Auth: `useAuth`
- Dashboard: `useDashboardSummary`
- Buildings: list, detail, form, services, meter readings, contract services
- Rooms: list, detail, form
- Tenants: list, detail, form
- Contracts: list, detail, form, occupants, payments, renewals, services, handover readings
- Billing: period list, period workspace, invoice actions
- Toasts: `useToast`

UI primitives live in `app/components/ui/**` and domain components live under their product folders. Billing has dedicated components for KPI strip, draft grid, mobile draft rows, discrepancy callouts, issue, payments, bulk payment, close, unissue, and audit.

## Auth And Permissions

- Supabase Auth provides sessions.
- User role is read from `app_metadata.role`.
- Supported roles: `admin`, `manager`.
- Route protection is in `app/middleware/auth.global.ts`; guest-only login behavior is in `app/middleware/guest.ts`.
- Server-side capability checks are centralized in `server/utils/permissions.ts`.
- `admin` has full CRUD plus `billing.close` and `billing.unissue`.
- `manager` has read/write operational capabilities but cannot close or unissue periods.

## Known Gaps / Not Yet Productized

- Tenant portal role and tenant self-service screens are not present.
- Email/notification flows are not wired into product actions.
- Resend, Turnstile, and GA config exist but are not central to the authenticated operations UI.
- Tiered electricity appears in rule helpers, but the main draft service currently blocks tiered electricity as unsupported in v1.
- CI pipeline is specified in OpenSpec, but no GitHub workflow file is currently present in the file map.
